    <article class="markdown-body entry-content" itemprop="text"><h1><a id="user-content-virtual-network-function-descriptor--schema" class="anchor" aria-hidden="true" href="#virtual-network-function-descriptor--schema"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Virtual Network Function Descriptor + Schema</h1>
<p>According to ETSI VNF [1], a VNF Descriptor (VNFD) is a deployment template which describes a VNF in terms of deployment and operational behaviour requirements. The VNFD also contains connectivity, interface and KPIs requirements that can be used by NFV-MANO functional blocks to establish appropriate Virtual Links within the NFVI between VNFC instances, or between a VNF instance and the endpoint interface to other Network Functions.</p>
<p>Our function descriptor schema specifies the content of a VNFD. It is based on the T-NOVA [2] flavor of the ETSI VNFD that can be found at [3]. It is, however, adapted and extended to meet the SONATA specific needs.</p>
<p>The schema is written in YAML based on JSON Schema [4] which can be easily tranlated to JSON, e.g. by using a json-to-yaml translator [5].</p>
<h2><a id="user-content-sections-of-the-function-descriptor" class="anchor" aria-hidden="true" href="#sections-of-the-function-descriptor"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Sections of the Function Descriptor</h2>
<p>Below we discuss the various section of a network function descriptor. The general descriptor section contains some of the manditory fields that have to be present in each and every function descriptor. All other sections might be optional.</p>
<h4><a id="user-content-general-descriptor-section" class="anchor" aria-hidden="true" href="#general-descriptor-section"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>General Descriptor Section</h4>
<p>At the root level, we first have the mandatory fields, that describe and identify the virtual network function in a unique way.</p>
<ul>
<li><strong>descriptor_schema</strong> referenced to the corresponding schema of a descriptor (e.g., URL or local path)</li>
<li><strong>$schema</strong> (optional) provides a link to the schema that is used to describe the network function and can be used to validate the VNF descriptor file. This is related to the original JSON schema specification.</li>
</ul>
<p>Moreover, the VNF signature, i.e the <em>vendor</em>, the <em>name</em>, and the <em>version</em>, is of great importance as it identifies the VNF uniquely.</p>
<ul>
<li><strong>vendor</strong> will identify the VNF uniquely across all VNF vendors. It should at least be comprised of the reverse domain name that is under your controll. Moreover, it might have as many sub-groups as needed. For example: eu.sonata-nfv.nec.</li>
<li><strong>name</strong> is the name of the VNF without its version. It can be created with any name written in lower letters and no strange symbols.</li>
<li><strong>version</strong> names the version of the VNF descritor. Any typical version with numbers and dots, such as 1.0, 1.1, and 1.0.1 is allows here. The VNF version must be increased with any new (changed) instance of the network function descriptor. Please note: The whole network function is composed of the descriptor and other artifacts, like virtual machine images. Thus, the network function may change, even if the description remains constant, just because another artifact changes. This might or might not be reflected in the version of the package descriptor.</li>
</ul>
<p>The general descriptor section also contains some optional components as outlined below.</p>
<ul>
<li><strong>author</strong> (optional) describes the author of the network function descriptor.</li>
<li><strong>description</strong> (optional) provides an arbitrary description of the VNF.</li>
</ul>
<h4><a id="user-content-virtual-deployment-units-section" class="anchor" aria-hidden="true" href="#virtual-deployment-units-section"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Virtual Deployment Units Section</h4>
<p>The virtual deployment unit section contains all the information regarding the VDUs, such as virtual machines an containers, that constitute the virtual network functions. The section is mandatory and starts with:</p>
<ul>
<li><strong>virtual_deployment_units</strong> contains all the virtual deployment units (VDUs) that are handled by the network function.</li>
</ul>
<p>This section has to have at least one item with the following information:</p>
<ul>
<li><strong>id</strong> represents a unique identifer within the scope of the VNF descriptor.</li>
<li><strong>vm_image</strong> (optional) specifies a reference to the virtual machine image (or container) that is used for the virtual network function. The image location can be a local file, a file within a package, a remove locatoin, that might be accessed via HTTP, or a reference within the SONATA service platform.</li>
<li><strong>vm_image_format</strong> (optional) specifies the image format, such as raw, vmdk, iso, and docker.</li>
<li><strong>vm_image_md5</strong> (optional) represent an MD5 hash of the virtual machine image. It is highly recommended to provide an MD5 hash, not only to verify the image, but to also make versioning of the whole virtual network function easier.</li>
<li><strong>resource_requirements</strong> details the resources required by the VDU even further.</li>
<li><strong>connection_points</strong> (optional) names the connection points offered by the VDU. The connection points can be used to interconnect various VDUs or to connect the VDU to an VNF connection point and to the outside world.</li>
<li><strong>monitoring_parameters</strong> (optional) names the monitoring parameters that are collected for this specific VDU and used, e.g. to trigger scaling operations.</li>
<li><strong>scale_in_out</strong> (optional) specifies the minimum and maximum number if VDU instances.</li>
</ul>
<h4><a id="user-content-connection-points-section" class="anchor" aria-hidden="true" href="#connection-points-section"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Connection Points Section</h4>
<ul>
<li><strong>connection_points</strong> (optional)</li>
</ul>
<p>While the parent section is optional, once it is specified it has to have at least one item with the following information:</p>
<ul>
<li><strong>id</strong> A VNF-unique id of a connection point which can be used for references.</li>
<li><strong>type</strong> The type of connection point, such as a virtual port, a virtual NIC address, a physical port, a physcial NIC address, or the endpoint of a VPN tunnel.</li>
<li><strong>virtual_link_reference</strong> (optional) (deprecated) A reference to a virtual link, i.e. the virtual_links:id.</li>
</ul>
<h4><a id="user-content-virtual-links-section" class="anchor" aria-hidden="true" href="#virtual-links-section"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Virtual Links Section</h4>
<ul>
<li><strong>virtual_links</strong> (optional) A VNF internal virtual link interconnects at least two connection points.</li>
</ul>
<p>While the parent section is optional, once it is specified it has to have at least some of the following information:</p>
<ul>
<li><strong>id</strong> A VNF-unique id of the virtual link which can be used for references.</li>
<li><strong>connectivity_type</strong> The connectivity type, such as point-to-point, point-to-multipoint, and multipoint-to-multipoint.</li>
<li><strong>connection_points_reference</strong> The references to the connection points connected to this virtual link.</li>
</ul>
<h4><a id="user-content-vnf-lifecycle-events-section" class="anchor" aria-hidden="true" href="#vnf-lifecycle-events-section"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>VNF Lifecycle Events Section</h4>
<ul>
<li><strong>lifecycle_events</strong> (optional) An array that contains VNF workflows for specific lifecycle events such as <em>start</em>, <em>stop</em>, <em>Scale_out</em>, <em>update</em>, etc.</li>
</ul>
<p>While the parent section is optional, once it is specified it has to have at least some of the following information:</p>
<ul>
<li><strong>vnf_container</strong> The VNF container that is associated with the lifecycle event.</li>
<li><strong>events</strong> The actual event such as start, stop, scale_out, update, etc.</li>
</ul>
<h4><a id="user-content-deployment-flavours-section" class="anchor" aria-hidden="true" href="#deployment-flavours-section"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Deployment Flavours Section</h4>
<ul>
<li><strong>deployment_flavour</strong> (optional) The flavours of the VNF that can be deployed.</li>
</ul>
<p>While the parent section is optional, once it is specified it has to have at least some of the following information:</p>
<ul>
<li><strong>flavour_key</strong> A VNF-unique id of the deployment flavour which can be used for references.</li>
<li><strong>vdu_reference</strong> A reference to the VDU.</li>
</ul>
